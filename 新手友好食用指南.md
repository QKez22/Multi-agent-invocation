## 2. 项目背景知识扫盲

### 2.1 什么是 Agent？

**简单理解**：Agent 就是一个能"自主思考、自主行动"的 AI 程序。

传统程序：
```
用户输入 → 程序处理 → 输出结果
（程序的每一步都是程序员写死的）
```

Agent 程序：
```
用户输入 → AI 思考 → 决定做什么 → 调用工具 → 继续思考 → ... → 输出结果
（AI 可以根据情况动态决定下一步做什么）
```

**举个例子**：
- 传统程序：用户问"什么是数据库索引"，程序直接从数据库查答案返回
- Agent 程序：用户问"什么是数据库索引"，AI 思考后决定：
  1. 先理解用户意图（是知识问答还是闲聊？）
  2. 改写问题（让搜索更准确）
  3. 检索知识库（找相关文档）
  4. 判断检索结果够不够（不够就追问用户）
  5. 生成回答（基于检索结果）

**本项目的 Agent**：就是一个智能知识库助手，能理解用户问题，自动决定怎么处理，最终给出准确回答。

---

### 2.2 什么是 RAG？

**RAG = Retrieval-Augmented Generation（检索增强生成）**

**为什么需要 RAG？**
- 大语言模型（LLM）的知识是训练时学到的，有截止日期
- LLM 不知道你公司的内部文档、产品手册
- 直接问 LLM，它可能"编造"答案（幻觉）

**RAG 怎么解决？**
```
用户问题
    ↓
① 检索：从你的知识库中找到相关文档
    ↓
② 增强：把找到的文档作为"参考资料"
    ↓
③ 生成：让 LLM 基于参考资料生成回答
```

**本项目的 RAG 流程**：
1. 用户问"数据库索引怎么优化"
2. 系统从知识库中搜索相关文档（向量相似度搜索）
3. 找到 5 篇相关文档
4. 把这 5 篇文档 + 用户问题一起发给 LLM
5. LLM 基于这些文档生成回答，并标注引用来源

---

### 2.3 什么是向量数据库？

**传统数据库**：存的是文字、数字，搜索靠精确匹配
```sql
SELECT * FROM articles WHERE title LIKE '%数据库索引%'
```

**向量数据库**：存的是"向量"（一串数字），搜索靠语义相似度
```
用户问题："数据库索引怎么优化"
    ↓ 转换成向量
[0.12, 0.85, 0.33, ...]
    ↓ 和知识库中的向量比较相似度
找到最相似的 5 篇文档
```

**为什么要用向量？**
- "数据库索引" 和 "索引优化技巧" 文字不同，但语义相似
- 传统搜索找不到，向量搜索能找到

**本项目用的向量数据库**：
- 学习阶段：FAISS（本地文件，零依赖）
- 生产环境：Milvus（分布式向量数据库）

---

### 2.4 什么是 LLM？

**LLM = Large Language Model（大语言模型）**

就是 ChatGPT、Claude 这类 AI 模型。本项目中，LLM 负责：
- 理解用户意图（是知识问答还是闲聊？）
- 改写问题（让搜索更准确）
- 判断检索结果够不够
- 生成最终回答

**本项目支持的 LLM**：
- OpenAI GPT 系列
- 国产大模型（通义千问、文心一言等）
- 本地部署的模型（Ollama）

---

### 2.5 什么是 Tool？

**Tool = 工具**，就是 Agent 可以调用的"能力"。

**类比**：Agent 像一个项目经理，Tool 像项目经理手下的员工
- 项目经理（Agent）负责思考、决策、分配任务
- 员工（Tool）负责具体执行，比如搜索文档、读取记忆、生成回答

**本项目的 Tool**：
```
Tool 名称              功能                    超时时间
───────────────────────────────────────────────────────
knowledge_search       知识库检索              30 秒
question_rewrite       问题改写                10 秒
rerank                 结果重排序              10 秒
answer_generation      答案生成                30 秒
memory_read            读取对话记忆            5 秒
memory_write           写入对话记忆            5 秒
ocr_extract            OCR 文档解析            60 秒
```

## 3. 环境准备与项目启动

### 3.2 项目启动步骤

**第一步：克隆项目**
```bash
git clone https://github.com/your-username/ai-knowledge-system.git
cd ai-knowledge-system
```

**第二步：启动数据库**
```bash

# 创建数据库，执行 sql/init.sql 初始化表结构
```

## 6. 核心概念详解

### 6.1 五层架构

```
Interface Layer（接口层）
├── 职责：接收 HTTP 请求，返回响应
├── 文件：api/routes.py, api/agent_routes.py
└── 类比：公司的前台，只负责接待，不负责处理业务

Orchestrator Layer（编排层）
├── 职责：协调整个 Agent 的执行流程
├── 文件：agent/orchestrator.py, agent/planner.py, agent/executor.py
└── 类比：项目经理，负责规划、分配任务、监督执行

Tool Layer（工具层）
├── 职责：提供具体的能力（检索、记忆、OCR 等）
├── 文件：tools/registry.py, tools/*.py
└── 类比：技术工人，各有所长，按需调用

Memory Layer（记忆层）
├── 职责：管理不同层级的记忆
├── 文件：tools/memory_read.py, tools/memory_write.py
└── 类比：公司的知识库和档案室

Evaluation Layer（评估层）
├── 职责：判断结果质量，决定是否继续
├── 文件：agent/planner.py (check_sufficiency)
└── 类比：质检部门，确保输出质量
```

**为什么要分层？**
- 每层职责单一，容易理解和维护
- 可以独立测试和替换（比如换向量数据库只改 Tool 层）
- 方便团队协作（不同人负责不同层）

---

### 6.2 多 Agent 协作

**Router Agent（路由 Agent）**
```
用户问题："什么是数据库索引"
    ↓
Router Agent 识别意图
    ↓
意图：知识问答
    ↓
分发给 Knowledge QA Agent
```

**Knowledge QA Agent（知识问答 Agent）**
```
接收问题
    ↓
改写问题 → "数据库索引的定义、分类和应用场景"
    ↓
检索知识库 → 找到 5 篇相关文档
    ↓
判断结果够不够 → 够了
    ↓
生成回答 → 基于检索结果生成详细回答
    ↓
返回给用户
```

**为什么要分多个 Agent？**
- 不同任务的处理逻辑差异很大（闲聊 2 秒，推理 15 秒）
- 分开后每个 Agent 只做一件事，可以独立优化
- 新增 Agent 很容易，不影响其他 Agent

### 6.3 四级记忆体系

**Layer 1: 短期记忆（AgentState）**
```
内容：当前对话的临时状态
存储：进程内存
生命周期：单次请求
速度：零网络开销
用途：步骤列表、中间结果、状态机
```

**Layer 2: 会话记忆（Redis）**
```
内容：最近 N 轮对话历史
存储：Redis
生命周期：会话级别（24 小时）
速度：毫秒级
用途：上下文连续性、多轮对话
```

**Layer 3: 用户记忆（MySQL）**
```
内容：用户偏好、历史问答
存储：MySQL
生命周期：永久
用途：个性化推荐、用户画像
```

**Layer 4: 知识记忆（向量库）**
```
内容：知识库文档的向量表示
存储：FAISS/Milvus
生命周期：随知识库更新
用途：语义相似度检索
```

**为什么要分四级？**
- 不同层级的读写速度和生命周期不同
- 短期记忆是进程内的零开销
- 会话记忆是 Redis 的毫秒级
- 用户记忆是 MySQL 的持久化
- 知识记忆是向量库的语义检索

**第三步：理解 Orchestrator 的 run() 方法**
```python
def run(self, input_text, ...):
    # 1. 创建状态
    state = self.create_state(input_text, ...)

    # 2. 输入验证
    if not self.policies.validate_input(input_text):
        return error_response

    # 3. 规划步骤
    planned_steps = self.planner.plan_steps(state)

    # 4. 逐步执行
    for step_name in planned_steps:
        step = state.add_step(step_type, step_name)
        self.event_bus.publish(StepStartedEvent(...))

        # 5. 带重试的执行
        while not step_done:
            try:
                self.executor.execute_step(state, step)
                step_done = True
            except Exception as e:
                if self.policies.should_retry(retry_count, e):
                    retry_count += 1
                else:
                    state.fail(str(e))

        # 6. 检查是否终止
        should_terminate, reason = self.planner.should_terminate(state)
```

### 7.2 跟着请求走

**一个知识问答请求的完整流程**：

```
1. 用户提问："什么是数据库索引"
   ↓
2. api/agent_routes.py 接收请求
   ↓
3. orchestrator.py 的 run() 方法开始执行
   ↓
4. planner.py 的 plan_steps() 规划步骤：
   - memory_read（读取记忆）
   - question_rewrite（改写问题）
   - knowledge_search（检索知识库）
   - sufficiency_check（判断结果够不够）
   - answer_generation（生成回答）
   - memory_write（写入记忆）
   ↓
5. executor.py 逐步执行每个步骤
   ↓
6. 每个步骤通过 tool_registry.invoke_tool() 调用工具
   ↓
7. 所有步骤执行完成，返回结果
```

### 7.3 关键代码片段解读

**Planner 怎么决定做什么？**
```python
def plan_steps(self, state):
    # 1. 识别意图
    intent = self.recognize_intent(state)

    # 2. 根据意图规划步骤
    if intent == IntentType.CHITCHAT:
        # 闲聊：只读记忆、生成回答、写记忆
        return ["memory_read", "answer_generation", "memory_write"]
    elif intent == IntentType.KNOWLEDGE_QA:
        # 知识问答：完整流程
        return ["memory_read", "question_rewrite", "knowledge_search",
                "sufficiency_check", "answer_generation", "memory_write"]
    elif intent == IntentType.ADMIN_OPERATION:
        # 管理操作：调用管理工具
        return ["admin_tool_call"]
```

**ToolRegistry 怎么调用工具？**
```python
def invoke_tool(self, tool_name, parameters, run_id):
    # 1. 验证参数
    tool.validate_input(parameters)

    # 2. 开始追踪
    tool_call_id = tracker.start_tool_call()

    # 3. 超时控制
    with ThreadPoolExecutor() as executor:
        future = executor.submit(tool.execute, parameters)
        result = future.result(timeout=tool.metadata.timeout_ms)

    # 4. 结束追踪
    tracker.end_tool_call(tool_call_id)

    return result
```
